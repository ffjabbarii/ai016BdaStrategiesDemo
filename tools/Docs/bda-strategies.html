BDA document scanning flows

This page explains the common ways to extract structured data from documents like W-2s and bank statements using Amazon Bedrock Data Automation (BDA) and Blueprints. The key idea: you typically create blueprints once and reuse them across many documents.

Flow A — Console “ad-hoc” (fast testing)

  • Create a BDA project and a blueprint in the AWS console.
  • Upload one document, run extraction, inspect the output.
  • Best for: learning, demoing, quick validation.

Flow B — Create once, reuse forever (recommended)

  • Create a blueprint for each document type (e.g., W-2, Bank Statement).
  • Attach blueprints to a project. Run the same project for many documents.
  • Best for: repeatable extraction, consistent outputs.

Flow C — API-first (no console in daily use)

  • Upload docs to S3 (or provide inputs per your integration).
  • Invoke BDA via API from a script/service.
  • Store JSON results in S3/DB for downstream processing.
  • Best for: automated apps, developer workflows, CI-style processing.

Flow D — Event-driven pipeline (production)

  • S3 put event triggers Lambda / Step Functions.
  • Pipeline runs BDA, then normalizes output, then stores results.
  • Optional: notify user, generate summary report, publish to a portal.
  • Best for: hands-off processing at scale.

Flow E — Batch + human review (high accuracy)

  • Batch ingest many docs (statements over many months).
  • Auto-extract, then apply validation rules.
  • Flag uncertain fields for review (human-in-the-loop).
  • Best for: compliance, high-confidence financial totals.

W-2 vs Bank statements

  • W-2: tends to be stable/standard → a single blueprint is often enough.
  • Bank statements: layouts vary by bank → you may need multiple blueprints or more flexible post-processing.
